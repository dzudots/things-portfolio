# Архитектура конфиденциальности «Стак»

## Принцип

Сервис — **личный домашний учёт имущества**, не реестр активов и не канал
налоговой/страховой отчётности. Цель хранения: показать пользователю его же
портфель. Не цель: профилирование для третьих сторон или «мониторинг для ФНС».

## Модель данных

```text
[Глобальный рынок]          [Приватный аккаунт]
comp_listings  ──┐          users (email, bcrypt)
canonical_models ┤          items (owner_id) ── encrypted: cost, notes, override
                 │          valuation_snapshots (только через owner)
                 └─ оценка ──► mid/low/high в снимках владельца
```

- **Comps** обезличены: модель + состояние + регион + цена. Нет user_id.
- **Портфель** изолирован `owner_id`; API/HTML всегда фильтруют по сессии.
- **Чувствительные поля** шифруются Fernet до записи в SQLite/Postgres.
- **Не храним**: ИНН, паспорт, VIN, серийные номера, улицу/дом.

## Ключи

- `THINGS_SECRET_KEY` — сессии / общее
- `THINGS_DATA_KEY` — Fernet для полей портфеля (обязателен в production)

Смена `THINGS_DATA_KEY` без миграции сделает старые ciphertext нечитаемыми.

## Права пользователя

1. Экспорт JSON (`GET /account/export`)
2. Удаление аккаунта и всех связанных строк (`POST /account/delete`)
3. Отключение алертов/дайджеста

## Ограничения (честно)

Шифрование at rest защищает от утечки файла БД «как есть», но **не** от
компрометации работающего сервера с ключом в памяти. Для максимальной
конфиденциальности следующий шаг — client-side encryption (ключ из пароля
пользователя, zero-knowledge). Это отдельный этап после product-market fit.

Юридически: при официальном запросе уполномоченного органа оператор может
быть обязан раскрыть данные — это не «фоновый налоговый мониторинг», а
отдельная процедура. Продукт не проектируется как инструмент уклонения
от закона.
